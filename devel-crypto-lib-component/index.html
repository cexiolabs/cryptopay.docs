    @cexiolabs/сrypto.component

    Crypto Wallet Component - High Level интеграция

    Under Hood

    Таблицы в БД

    • component - идентификатор инстанса CryptoWallet
    • cex_wtf2pss - конфигурация CexCryptoWallet по каждой валюте. (CexCryptoWallet - это набор из Wtf2Client-ов. Строках этой таблицы = инстансу Wtf2Client)
    • callback_unexpected_deposit - здесь записываются все коллбеки о депозитах, пришедшие от CryptoWallet которые не соответствуют нашим данным. Аудит раcсинхрониизации.
    • callback_unexpected_withdrawal - здесь записываются все коллбеки о транзакции на вывод, пришедшие от CryptoWallet которые не соответствуют нашим данным. Аудит раcсинхрониизации.
    • address_declaration - заготовка/декларация намерения пойти в CryptoWallet за новым адресом. Этот guard нужен на случай падения после запроса, но до ответа из CryptoWallet. По-сути тут хранятся наш клиентский идентификатор для запроса в CryptoWallet.
    • address_pool - хранит адреса которыми мы владеем, это и есть пул адресов.
    • address_bound - помечает адрес как использованный для пользовательского addressClientId. Запись в эту таблицу использует адрес из пула.
    • deposit - информация о депозите
    • deposit_confirmation - история коллбеков по депозиту
    • withdrawal - базовая информация о транзакции на вывод
    • withdrawal_peer - дополнительная информация по peer валюте
    • withdrawal_utxo - дополнительная информация по utxo валюте
    • withdrawal_utxo_io - список проводок которые нужно включить в utxo-транзакцию ( pss-provider-wtf2 пока не понятно как делать групповые транзакции)
    • withdrawal_confirmation - история коллбеков по транзакции на вывод

    Функции(процедуры) в БД

    • sp_withdrawal_state - возвращает состояние транзакции на вывод (скрывает сложность вычисления статуса по аккаунтингу)

    Состояния депозита

    Выражается через аккаунтинг

    • PENDING - если есть только проводка pending_transaction_id
    • ACCEPTED - если есть проводка pending_transaction_id + проводка на INT счет привязанный к адресу
    • REJECTED - если есть проводка pending_transaction_id + проводка на EXT счет привязанный к адресу (вернулись туда, откуда пришли)

    Генерация нового адреса

    Процесс получения депозитного адреса следующий:

    1. Пользователь вызывает generateNewAddress(...) в личной SQL транзакции.
    2. Пытаемся взять готовый адрес из пула адресов
    3. Запускаем отложенный шедулер _scheduleUpdateAddressPool, который позже запросит реально новые адреса для заполнения пула.
    4. Если адреса в пуле нет, то выполняем запрос в CryptoWallet в синхронном режиме для получения нового адреса. Тут действия происходят в отдельной SQL транзакции, в несколько попыток и если пользовательская транзакция будет отменена, то созданный адрес не будет утерян, а просто сохранится в пул адресов.
    5. Полученный адрес биндим к пользовательскому addressClientId записью в таблицу own_address_bound (если пользователь отменит транзакцию, адрес "вернется" в пул)

    Метод _generateNewAddress(...) получает реально новый адрес в CryptoWallet. Делает следующее (с защитой от дабл):

    1. Создает UUID который используется как addressClientId в CryptoWallet и как будущий accountUuid
    2. В отдельной SQL транзакции выполняет запись о намерении создать адрес в таблицу address_declaration
    3. В следующей SQL транзакции выполняет:
      1. Блокировку записи из address_declaration через SELECT ... FOR UPDATE
      2. Создает два asset-а: EXT:${accountUuid} и INT:${accountUuid} (будущие депозиты отражаются как проводки EXT:${accountUuid} -> INT:${accountUuid})
      3. Запрашивает новый адрес у CryptoWallet#generateNewAddress(...)
      4. Информацию по сформированному адресу записывает в таблицу address_pool
      5. В этом месте новый адрес читается как добавленный в пул адресов.
      6. В случае падения этой SQL транзакции, у нас на память остается запись в таблице address_declaration, которая дает возможность повторно запросить информацию об адресе в CryptoWallet (но это пока не реализуется)

    Deposit-ы

    Движение средств по аккаунтингу выполняемые CryptoWalletComponent-ом:

    • EXT:xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx - Cчет источник, с которого расходуются деньги. Этот счет ВСЕГДА , в минусовой нотации, отображает РЕАЛЬНЫЙ баланс адреса в блокчейне. Где xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx - UUID адреса зарегистрированного в таблице address_declaration.
    • DEPOSIT_PROCESSING:tx_hash - счет который хранит деньги во время ожидания подтверждения депозита (набор необходимого колличества подтверждений или через isTrustedTransaction(...))
    • INT:xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx - Cчет назначения, куда приходят деньги. Это счет мерчанта, с ВИРТУЛЬНЫМИ деньгами, которые мерчант может использовать для операций внутри системы. Где xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx - UUID адреса зарегистрированного в таблице address_declaration.
    graph LR; EXT:xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx -- pending --> DEPOSIT_PROCESSING:tx_hash; DEPOSIT_PROCESSING:tx_hash -- rejected --> EXT:xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx; DEPOSIT_PROCESSING:tx_hash -- accepted --> INT:xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx;
    • xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx - UUID адреса зарегистрированного в таблице address_declaration
    • tx_hash - это хеш транзакции блокчейна (специфичен под каждый блокчейн, пример для TRX: 8C14FDDEF51E3545CF3AEEBBC08B7D7AA76F8924C2FAFD6F5890489FFBFCB2F6)

    Идентификатор проводок

    Во всех проводках по депозиту, в качестве идентификатора name, используется tx_hash. Согласно ограничениям в базе аккаунтинга, проводки (вход и выход по счету) могут иметь одинаковый идентификатор name, только в разрезе одного счета (не возможно дважды выполнить депозит, по одному счету, с одинаковым tx_hash). Это свойство аккаунтинга позволяет провести длинную операцию по аккантингу, используя единый идентификатор, и защищает от разветвлений (только одна цепочка прохождения).

    Обработка deposit-коллбеков от CryptoWallet

    1. Получаем коллбек от CryptoWallet
    2. SQL транзакция: Ищем связанный с коллбеком адрес ( если нет связанного адреса, сохраняем данные коллбека в таблицу callback_unexpected_deposit для будущего расследования)
    3. SQL транзакция: Ищем связанную с коллбеком PENDING аккаунтинг-проводку
      1. Если проводки нет, это значит, что получили первый коллбек (но не обязательно 0-й конфирмейшен)
        1. SQL транзакция: Создаем PENDING аккаунтинг-проводку
        2. SQL транзакция: Создаем запись в таблице deposit
        3. SQL транзакция: Создаем запись в таблице deposit_confirmation
        4. Добавляем в очередь коллбек для пользователя - PENDING (isFirstEvent: true)
        5. Если количество конфирмейшенов удовлетворяет минимуму для принятия OR isTrustedTransaction(...) вернул true
          1. SQL транзакция: Финализируем ACCEPTED аккаунтинг-проводку
          2. Добавляем в очередь коллбек для пользователя - ACCEPTED
      2. Если проводка есть, это значит, что получили коллбек с конфирмейшеном
        1. SQL транзакция: Создаем запись в таблице deposit_confirmation
        2. SQL транзакция: Если аккаунтинг-проводка == PENDING
          1. Если колличество конфирмейшенов удовлетворяет минимуму для принятия OR isTrustedTransaction(...) вернул true
            1. SQL транзакция: Финализируем ACCEPTED аккаунтинг-проводку
            2. Добавляем в очередь коллбек для пользователя - ACCEPTED
          2. Иначе, Добавляем в очередь коллбек для пользователя - PENDING
    4. SQL транзакция: COMMIT
    5. Отправляем пользователю очередь коллбеков

    Withdrawal-ы

    Движение средств по аккаунтингу выполняемые CryptoWalletComponent-ом, где xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx - UUID withdrawal_uuid в таблице withdrawal:

    • WITHDRAWAL_SOURCE - счет источник, с которого расходуются деньги
    • WITHDRAWAL_APPROVING:xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx - счет который хранит деньги во время ожидания подтверждения операции пользователем
    • WITHDRAWAL_PROCESSING:xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx - счет который хранит деньги во время ожидания выполнения транзакции в бэкенде CryptoWallet
    • WITHDRAWAL_TARGET - счет назначения, куда приходя деньги по завершенной выплате (типо блокчейн счет)
    • WITHDRAWAL_FEE - счет для списания комиссии сети. Размер комиссии, в ИНВЕРТИРОВАННОМ виде, переводится с этого счета на счет WITHDRAWAL_SOURCE (для сохранения единого withdrawal_uuid в именах проводок, т.к. не возможно сделать второй вывод из WITHDRAWAL_SOURCE с тем же withdrawal_uuid)
    graph LR; WITHDRAWAL_SOURCE -- pending-approve --> WITHDRAWAL_APPROVING; WITHDRAWAL_APPROVING -- user-cancel --> WITHDRAWAL_SOURCE; WITHDRAWAL_APPROVING -- pending-processing --> WITHDRAWAL_PROCESSING; WITHDRAWAL_PROCESSING -- rejected --> WITHDRAWAL_SOURCE; WITHDRAWAL_PROCESSING -- accepted --> WITHDRAWAL_TARGET; WITHDRAWAL_FEE -- accepted-fee --> WITHDRAWAL_SOURCE;

    Идентификатор проводок

    Во всех проводках по выводу, в качестве идентификатора name, используется один и тот же uuid (идентификатор транзакции на вывод). Согласно ограничениям в базе аккаунтинга, проводки (вход и выход по счету) могут иметь одинаковый идентификатор name, только в разрезе одного счета (не возможно дважды выполнить вывод, по одному счету, с одинаковым uuid). Это свойство аккаунтинга позволяет провести длинную операцию по аккантингу, используя единый идентификатор, и защищает от разветвлений (только одна цепочка прохождения).

    UTXO Multi Address Withdrawal

    В случае валюты utxo, входящих(источников) и исходящий(назначение) адресов может быть много. Для каждого входящего(источника) есть WITHDRAWAL_SOURCE счет, и схема на диаграмме размножается на многие-ко-многим, но обрабатываться атомарно всех группой.

    Регистрация транзакции на вывод

    1. Процессинг вызывает метод registerTransactionUtxo/registerTransactionPeer и передает параметром свою SQL транзакцию. Все действия происходят в SQL транзакции процессинга.
    2. Регистрируем транзакцию у CryptoWallet через registerTransactionUtxo/registerTransactionPeer. CryptoWallet возвращает специальный секретный токен, который нужен для второго шага Подтверждение транзакции на вывод, и withdrawal_uuid.
    3. Переводим деньги в аккаунтинге, создаем pending-approve-проводку(ки, для utxo их может быть много) на счет WITHDRAWAL_APPROVING.
    4. Проверяем баланс исходящего(-щих) счета(-ов) WITHDRAWAL_SOURCE, что они не ушли в минус (больше равно нуля). Здесь овердрафта быть НЕ может.
    5. Регистрируем вывод крипты в таблице withdrawal
    6. Записываем данные по pending-approve проводкам в таблицах withdrawal_peer или withdrawal_utxo + withdrawal_utxo_io в зависимости от типа валюты.

    7. Возвращаем данные о созданном выводе крипты. Конец. ВАЖНО: Процессинг НЕ должен знать секретный токен полученный от CryptoWallet, это приватная информация CryptoWalletComponent-а.

      Падение SQL транзакция процессинга

      Падение SQL транзакция процессинга не оставляет никаких артефактов в базе, в том числе баланс возвращается на исходные цифры. В бэкенде CryptoWallet остается на память зарегистрированная транзакция, но она никогда не будет подтверждена, т.к. секретный токен подтверждения, полученный от CryptoWallet, безвозвратно утерян. Подразумевается, что бэкенд CryptoWallet-та спустя таймаут, удалит(или пометит как expired) эту регистрацию на своей стороне.

      CexCryptoWallet/pss-provider-wtf2

      Имплементация CexCryptoWallet/pss-provider-wtf2 фактически ничего не отправляет, т. к. pss-provider-wtf2 делает вывод крипты в одно действие без подтверждения, используя методику с Client ID.

    Подтверждение транзакции на вывод

    1. Процессинг вызывает метод confirmTransactionUtxo/confirmTransactionPeer. Все действия происходят в личной SQL транзакции компонента (процессинг не может отменить модификации).
    2. SQL транзакция
      1. Проверяем на дубликат запроса, что счет WITHDRAWAL_PROCESSING еще не пополнен.
      2. Если деньги еще не поступили, выполняем:
        1. Создаем pending-processing-проводку(ки, для utxo их может быть много) на счет WITHDRAWAL_PROCESSING. ВАЖНО: Наличие этой проводки делает невозможным Отмена транзакции на вывод.
        2. Проверяем баланс исходящего(-щих) счета(-ов) WITHDRAWAL_APPROVING, что они равны нулю.
      3. COMMIT SQL транзакция. ВАЖНО: Если не выполнить COMMIT появляется возможность двойной траты (например запрос ушел в CryptoWallet, но оборвался на обратном пути, деньги ушли в блокчейн и вернулись назад мерчанту по аккаунтингу)

    3. Подтверждаем транзакцию у CryptoWallet через confirmTransactionUtxo/confirmTransactionPeer используя специальный секретный токен полученных на шаге Регистрация транзакции на вывод. Тут может быть какая нибудь retry-йлка.

      Повторы/идемпотентность

      Эта последовательность защищена от многократного выполнения. Деньги будут быть переведены только в первый запрос. Подтверждение транзакции в бэкенд CryptoWallet всегда возвращает результат первого запроса.

      Фоновая обработка

      По событиям(коллбекам) от CryptoWallet определяем результаты обработки транзакции на бэкенде CryptoWallet-а. По успешным признаками - финализируем accepted-проводку(ки, для utxo их может быть много) доставляя результат на счет WITHDRAWAL_TARGET. Если транзакция была отреджекчена на бэкенде CryptoWallet-а, проводим полный реджект по нашему аккаунтингу через rejected-проводку(-ки), возвращая средства на WITHDRAWAL_SOURCE.

    Обработка withdrawal-коллбеков от CryptoWallet

    1. Получаем коллбек от CryptoWallet
    2. SQL транзакция: BEGIN
      1. Если CryptoWalletComponent выключен, отвечаем на коллбек ошибкой. КОНЕЦ
      2. Ищем связанный с коллбеком withdrawal_uuid ( если нет связанного withdrawal_uuid, сохраняем данные коллбека в таблицу callback_unexpected_withdrawal для будущего расследования). КОНЕЦ
      3. Записываем информацию о коллбеке в таблицу withdrawal_confirmation
      4. Если получили Status.REJECTED или Status.CANCELLED
        1. Если accepted аккаунтинг-проводка есть
          1. Оповещение системы мониторинга: CRITIAL_ALERT (взлом бэкенда CryptoWallet?)
          2. Отключение (блокировка) CryptoWalletComponent вызовом SQL функции sp_block_component(...)
        2. Иначе
          1. Если reject аккаунтинг-проводка еще нет
            1. Создаем rejected аккаунтинг-проводку, c примечанием Cancel/Reject by CryptoWallet (перевод денег WITHDRAWAL_PROCESSING -> WITHDRAWAL_SOURCE)
            2. Проверяем баланс исходящего(-щих) счета(-ов) WITHDRAWAL_PROCESSING, что они равны нулю.
            3. Запоминаем коллбек для пользователя - REJECTED
      5. Если получили Status.ACCEPTED
        1. Если rejected аккаунтинг-проводка есть
          1. Оповещение системы мониторинга: CRITIAL_ALERT (взлом бэкенда CryptoWallet?)
          2. Отключение (блокировка) CryptoWalletComponent вызовом SQL функции sp_block_component(...)
        2. Иначе
          1. Если accepted аккаунтинг-проводка еще нет
            1. Создаем accepted аккаунтинг-проводку (перевод денег WITHDRAWAL_PROCESSING -> WITHDRAWAL_TARGET)
            2. Проверяем баланс исходящего(-щих) счета(-ов) WITHDRAWAL_PROCESSING, что они равны нулю.
            3. Создаем accepted-fee аккаунтинг-проводку В инвертированной нотации (с минусом) (перевод денег WITHDRAWAL_FEE -> WITHDRAWAL_SOURCE)
            4. Запоминаем коллбек для пользователя - ACCEPTED
    3. SQL транзакция: COMMIT
    4. Отправляем пользователю коллбек (если есть)

    Отмена транзакции на вывод

    1. Процессинг вызывает метод confirmTransactionUtxo/confirmTransactionPeer. Все действия происходят в личной SQL транзакции компонента (процессинг не может отменить модификации).
    2. SQL транзакция
      1. Проверяем, что счет WITHDRAWAL_APPROVING еще не израсходован в сторону WITHDRAWAL_PROCESSING.
        1. заканчиваем успешно user-cancel-проводку(-ки). Деньги возвращаются на счет(-а) WITHDRAWAL_SOURCE. ВАЖНО: Наличие этой проводки делает невозможным выполнение Подтверждение транзакции на вывод.
        2. Проверяем баланс исходящего(-щих) счета(-ов) WITHDRAWAL_APPROVING, что они равны нулю.
